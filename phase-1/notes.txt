Why permissions exist in the first place? 
The linux system can have many users. Permissions is how the system decides who is allowed to do what to do what to which files. Without permissions, any user on the mission could read, modify or delete someone else's file, including system files.

When you type https://github.com into your browser and press Enter — what actually happens before the page appears?
1. The browser needs to find Github's address, it will look up the ip address of github.com
2. The browser opens a connection to that address. A Transmission Control Protocol allows the communication between the browser and the server to be established.
3. Because it is HTTPS, they do a security handshake. A Transport Layer Security handshake, this means that the browser and the server agree on how to encrypt the communication so nobody in between can read it.
4. The browser sends a HTTP request to the server, in this case, it'll be a GET request
5. Github's server sends back a HTTP response, the response includes a status code indicating what has occured. (e.g 200 -OK, 500 - Server error)
6. The browser renders the page. It receives the html and shows the page on the browser

When you search for https://github.com, the browser converts it into numbers since the computers cannot understand human language. The browser on your screen, it then talks to the server where github.com is, after it receives a response, that means a channel or connection will be made. They will then agree to encrypt (secure, for it to be a secret) the data so as to not be read in between. The browser sends a request telling the server as to what it wants, the server will then respond back with what has occurred, whether it has succeeded or not. The browser then finally shows what you see on the screen, the github landing page.

What does the with statement do? Why is it better than just calling open() directly? What would happen if you deleted content.txt and then hit the / route?
The with statement enables the file that has been read from to be properly closed after being read from. If open() is just called, the file that is read from will only remain open and will not be closed, unless we add an f.close() property at the end. If content.txt is deleted and the I happen to hit the / route, it will return a FileNotFoundError: No such file or directory: 'content.txt' error.
